# TeleKG-Agent

A research prototype exploring three ideas for AI in telecom networks. Built solo, no GPU, runs on synthetic data.

## What's in here, in plain terms

**1. Smart test selection.** When a telecom network changes (new software, config tweak, etc.), instead of re-running every test in the suite, look up a knowledge graph that knows which features touch which KPIs, and only run the tests that are actually relevant. In our demo this cuts the test suite by ~72% while still catching everything important.

**2. KPI translator.** A network engineer can type something like "what fraction of calls succeed?" in plain English, and the system looks up the real underlying counters (RRC_Setup_Success, RRC_Setup_Attempt, etc.) and builds the correct formula automatically — instead of every team hardcoding their own KPI definitions.

**3. Signal-native pretraining experiment.** Most "telecom AI" models today are trained on *text* (specs, documents, tickets). This is a small experiment testing whether pretraining a model directly on *KPI numbers* (not text) gives it a head start on tasks like anomaly detection. Small-scale, but the result was clear: yes, it helps.

None of this uses real operator data — everything is synthetic, generated by the code in this repo.

## How it's built

```
Change happens on the network (e.g. software upgrade)
        │
        ▼
Knowledge Graph (FalkorDB)
  - which cells exist, how they're connected
  - which KPIs affect which other KPIs (e.g. high PRB usage → hurts Throughput)
  - which features touch which KPIs, which tests cover which KPIs
        │
        ▼
5 small agents take turns:
  1. Log the change
  2. Look up what's affected (graph lookup, no AI needed)
  3. Explain root cause (AI writes the explanation)
  4. Generate the actual test code (AI writes the code)
  5. Queue tests for execution
        │
        ▼
A short list of the exact tests that matter — not all 18, just the 5 that do
```

The KPI translator is a separate piece: it takes a free-text question, finds the real counters it should use (this is the "RAG" part — Retrieval Augmented Generation), and only lets the AI pick from real counters, never invented ones.

The signal-pretraining piece is fully separate — a small AI model trained directly on number sequences (KPI time-series) instead of words.

## Quick Start

```bash
git clone <this repo>
cd telekg-agent
pip install -r requirements.txt --break-system-packages
```

(If you get an "externally-managed-environment" error on Ubuntu/Debian, that `--break-system-packages` flag is the fix — see note at the bottom.)

### See it work, zero setup

```bash
python3 main.py --dry-run --eval
```

No API key, no database, no internet needed. Runs the full pipeline against fake/mock data so you can see the shape of the system in a few seconds.

### With a real AI model (Groq — free, fast, no GPU needed)

```bash
export GROQ_API_KEY=gsk_your_key_here   # get one free at https://console.groq.com
pip install groq --break-system-packages
python3 main.py --eval
```

### With a real graph database

```bash
docker run -p 6379:6379 falkordb/falkordb:latest
pip install falkordb --break-system-packages
python3 main.py --eval
```

### KPI translator benchmark

```bash
pip install sentence-transformers --break-system-packages
python3 run_kpi_eval.py --json results.json
```

### Signal-native pretraining experiment

```bash
pip install torch --index-url https://download.pytorch.org/whl/cpu --break-system-packages
python3 run_telesignal_poc.py --n-cells 60 --n-timesteps 1500
```

(Use that `--index-url` — it skips downloading several GB of unnecessary CUDA/GPU files you don't need on a CPU-only machine.)

## Repository Map

```
telekg/
  schema.py            — defines the graph: what node and edge types exist
  pm_registry.py        — the real counters (e.g. RRC_SETUP_SUCCESS) and KPI formulas built from them
  pm_simulator.py        — generates fake but realistic counter data
  simulator.py           — generates the fake network (100 cells, change events, etc.)
  graph_builder.py       — loads everything into the FalkorDB graph
  reasoner.py            — the lookup functions agents call ("what's affected by this release?")
  kpi_translator.py      — turns plain English into a KPI formula (the RAG piece)
  kpi_eval.py             — a 17-question test to check the translator's accuracy
  telesignal_model.py    — the small AI model for the signal-pretraining experiment
agents/
  pipeline.py             — the 5 agents, and the connection to Groq
main.py                   — run this to see the test-selection demo
run_kpi_eval.py            — run this to test the KPI translator
run_telesignal_poc.py      — run this to test signal-native pretraining
```

## What the results actually show

**Test selection:** out of 18 total tests, the system correctly picks the 5 relevant ones for a given change, and gets 100% of the critical tests right (no important test is skipped).

**KPI translator:** roughly 2 out of 3 plain-English KPI questions get matched to the right answer correctly, and it never makes up a fake counter name (validated 100% of the time — every counter it uses is checked against the real list). It's not perfect — sometimes it picks a similar-sounding KPI instead of the exact right one, especially on trickier phrasing (uplink vs downlink, "successful releases" vs "drops"). That's an honest limitation, not hidden.

**Signal-native pretraining:** a small AI model (110,864 parameters — tiny by AI standards) that's pretrained on raw KPI numbers first, then fine-tuned for anomaly detection, clearly beats one trained from scratch with no pretraining (F1 score 0.98 vs 0.66). This took about 2 minutes on a regular CPU, no GPU.

## Honest Limitations

- Everything is synthetic data. No real telecom operator data was used anywhere.
- The KPI translator gets about 2/3 of plain-English questions exactly right — good enough to demonstrate the idea, not production-ready.
- The signal-pretraining model is intentionally small (no GPU access). It proves the idea works at small scale, it's not a competitor to any large industrial model.

## Installing Python Packages on Ubuntu/Debian (WSL included)

Modern Ubuntu blocks plain `pip install` system-wide to protect the OS's own Python tools. The simplest fix, if you don't want to deal with virtual environments:

```bash
pip install <package> --break-system-packages
```

To stop typing that every time:

```bash
mkdir -p ~/.config/pip
echo -e "[global]\nbreak-system-packages = true" >> ~/.config/pip/pip.conf
```

After that, plain `pip install X` always works.
